# PART 1 ‚Äî Unit Semantics Must Be Explicit (Rate vs Total)

## Problem
Cost math requires knowing whether the entered amount is:
- rate per acre (lb/ac, oz/ac, gal/ac), or
- total product (lb, oz, gal), or
- time (hrs), or
- program item ($/acre)

Right now, the UI may only expose units like: GAL/LB/OZ/HRS, which is ambiguous.

## Solution (Best)
### A) Add explicit ‚Äú/AC‚Äù units to the unit dropdown
In `Log.tsx`, update the unit options:
- For SPRAY action: `OZ/AC`, `LB/AC`, `GAL/AC`, plus optionally `OZ`, `LB`, `GAL`
- For FERT action: `LB/AC`, `GAL/AC`, plus optionally `LB`, `GAL`
- For IRRIGATE: `HRS` (only)
- For OTHER/program: `$/AC` (optional, used for mating disruption)

Default selection:
- SPRAY and FERT default to `/AC` units
- IRRIGATE defaults to `HRS`

### B) If you cannot add /AC units (fallback)

- Label: **Per Acre**
- Default ON for SPRAY + FERT
- Default OFF for IRRIGATE
When ON, treat the entered amount as ‚Äúper acre‚Äù even if the unit lacks `/AC`.

# PART 3 ‚Äî Unit Normalization Guardrails (Avoid Silent Wrong Totals)

## Problem
Chemical Library units include mixed pricing units: GAL, LB, OZ, TON, ACRE, pack sizes like 2.5GAL/50LB.
If the Chemical unit doesn‚Äôt match the Log unit, you can produce wrong costs.

## Prototype-safe rule (Must Implement)
**Cost Engine v1 only calculates cost when units are compatible.**
If incompatible:
- `estimatedCost = 0`
- set a flag on the ChemicalApp: `costStatus = "UNIT_MISMATCH"` (see below)
- show small badge: ‚ÄúCost not estimated (unit mismatch)‚Äù

### Compatibility matrix
Define a function:

`areUnitsCompatible(chemicalUnit: string, logUnit: string): boolean`

Allowed matches (case-insensitive):
- Same base unit:
  - Chemical: GAL matches Log: GAL or GAL/AC
  - Chemical: LB matches Log: LB or LB/AC
  - Chemical: OZ matches Log: OZ or OZ/AC
  - Chemical: ACRE matches Log: $/AC (program items)
- TON only matches TON (v1)
  - If chemical unit is TON and log unit is GAL/AC or GAL ‚Üí mismatch (no density conversion in v1)
- Pack sizes (e.g., 2.5GAL, 50LB, 25LB, 7.5LB, 45OZ):
  - v1 behavior: treat as mismatch unless log unit matches that exact pack size string
  - OR simpler: convert the chemical‚Äôs unit to its base unit by stripping pack sizes:
    - 2.5GAL ‚Üí base GAL
    - 50LB / 25LB / 7.5LB ‚Üí base LB
    - 45OZ / 20OZ ‚Üí base OZ
  Choose one approach:
  - Preferred: **strip to base unit** so your pricing still works.

Implement helper:
`normalizeUnit(u: string): { base: "GAL"|"LB"|"OZ"|"TON"|"ACRE"| "HRS" | "USD_AC" | "UNKNOWN"; multiplier?: number }`

Examples:
- "2.5GAL" ‚Üí base GAL, multiplier 2.5
- "7.5LB" ‚Üí base LB, multiplier 7.5
- "45OZ" ‚Üí base OZ, multiplier 45
- "GAL/AC" ‚Üí base GAL (log-side), indicates per-acre
- "$/AC" ‚Üí base USD_AC (log-side program)
- "HRS" ‚Üí base HRS (no cost calc here unless you add water pricing later)

Then:
- Compare normalized bases for compatibility.
